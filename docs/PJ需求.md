# memxlife  
## MLSYS 课程项目  

随着代理时代（Agent Era）的到来，系统工程的格局正在经历一场根本性的范式转变。传统上，算子优化和高性能 GPU 调优等任务仅限于一小部分精英专家，他们具备从高层算法到 GPU 硬件架构细节的深厚垂直知识。在过去，大规模完成此类复杂项目需要数千人的团队。

### 第一阶段：GPU 性能分析指南 —— 使用 ncu 识别性能瓶颈  
**截止时间：2026 年 4 月 21 日上午 8 点**

NVIDIA Nsight Compute（ncu）是一款用于 CUDA 内核的交互式内核级性能分析工具。与观察全局时间线的 Nsight Systems（nsys）不同，ncu 深入每个内核的内部执行过程，展示硬件资源的消耗情况。

在本项目中，你的代理需要分析 ncu 输出的指标，以确定某个算子（例如矩阵乘法）的性能瓶颈。以下是核心指标类别及其在性能调优中的意义。

#### 1.1 核心概览指标  
在进行详细分析之前，首先使用 Roofline 模型判断该算子是计算受限（Compute-Bound）还是内存受限（Memory-Bound）。

- **sm__throughput.avg.pct_of_peak_sustained_elapsed**（计算利用率）  
  含义：流式多处理器（SM, Streaming Multiprocessor）计算单元的利用率。  
  分析：若该值较高，表明 GPU 的计算核心已充分负载。

- **gpu__compute_memory_throughput.avg.pct_of_peak_sustained_elapsed**（内存带宽利用率）  
  含义：GPU 内存带宽的利用率。  
  分析：若该值较高，说明数据传输速度正在限制性能。

- **SOL（Speed of Light）**：ncu 报告中常提到的“SOL Compute”和“SOL Memory”，表示当前算子达到了硬件理论峰值性能的百分比。

#### 1.2 内存层次结构指标  
如果算子属于内存密集型，则需识别存储层次结构中哪一层是瓶颈。

- **l1tex__t_sectors_pipe_lsu_mem_global_op_ld.sum**（L1 缓存）  
  分析：L1 缓存命中率与吞吐量。频繁访问全局内存且未命中 L1 缓存会显著增加延迟。

- **l2__throughput.avg.pct_of_peak_sustained_elapsed**（L2 缓存）  
  分析：L2 缓存作为 VRAM 与 SM 之间的桥梁。极高的 L2 利用率表明这两层之间存在大量数据交换。

- **dram__throughput.avg.pct_of_peak_sustained_elapsed**（VRAM/DRAM）  
  分析：外部 VRAM 带宽。若此值超过 80%，则代码可能需要更好的数据复用（例如利用共享内存 Shared Memory）。

#### 1.3 计算单元指标  
如果算子属于计算受限，则需识别具体哪些计算单元处于活跃状态。

- **sm__pipe_tensor_op_hmma_cycle_active.avg.pct_of_peak_sustained_active**（张量核心 Tensor Cores）  
  分析：至关重要！对于深度学习中的矩阵乘法，若该指标较低，意味着未能有效利用张量核心。

- **sm__pipe_fma_cycles_active.avg.pct_of_peak_sustained_active**（FP32/FMA）  
  分析：传统单精度浮点运算单元的利用率。

- **sm__sass_thread_inst_executed_op_fp32_pred_on.sum**  
  分析：实际执行的 FP32 指令总数。

#### 1.4 占用率与调度  
有时硬件利用率低是因为线程未能“填满”GPU。

- **sm__maximum_warps_per_active_cycle_pct**（理论占用率）  
  含义：基于寄存器和共享内存使用情况，硬件理论上可同时运行的最大 warp 百分比。

- **sm__warps_active.avg.pct_of_peak_sustained_active**（实际达到的占用率）  
  含义：算子执行期间实际达到的占用率。

- 分析：若实际占用率远低于理论占用率，表明存在指令延迟问题或线程块分布不均。

#### 1.5 常见性能瓶颈及诊断表

| 编号 | 瓶颈类型         | 关键指标                                                                 | 优化方向                                       |
|------|------------------|--------------------------------------------------------------------------|------------------------------------------------|
| 1    | VRAM 受限        | `dram__throughput > 70%`                                                 | 减少内存访问；提高数据复用；使用共享内存       |
| 2    | 计算受限         | 高 `sm__throughput`，高 `tensor_op_hmma`                                 | 考虑算法改进或降低精度（如 FP32 转 FP16/BF16） |
| 3    | 非合并访问       | `l1tex__t_sectors_pipe_lsu_mem_global_op_ld` 过高                         | 检查内存访问模式；确保相邻线程访问相邻地址     |
| 4    | Warp 分歧        | `sm__sass_thread_inst_executed_per_inst_executed < 32`                   | 减少 if/else 分支；确保 warp 内线程路径一致    |
| 5    | 存储体冲突       | `l1tex__data_bank_conflicts_pipe_lsu.sum > 0`                            | 调整共享内存索引（例如使用填充 Padding）       |

#### 1.6 给学生的建议：如何让你的代理使用这些数据？

**步骤 1：获取 Roofline 模型信息**  
让代理首先读取 `sm__throughput` 和 `gpu__compute_memory_throughput` 两个指标。

**步骤 2：特征分类**  
- 若内存利用率 % > 计算利用率 %，深入分析 `dram` 以及 `l1/l2` 相关指标。  
- 若计算利用率 % > 内存利用率 %，检查是否触发了 `tensor_op`。

**步骤 3：查找异常**  
检查占用率是否过低，或是否存在存储体冲突。

**步骤 4：映射回代码**  
将这些指标与你的 CUDA 内核代码关联起来（例如：循环展开是否不足？是否缺少 `__shared__` 数组？）。

##### 如何通过命令行获取这些指标？

```shell
# 示例：获取矩阵乘法内核的所有详细指标
ncu --metrics sm__throughput.avg.pct_of_peak_sustained_elapsed,gpu__compute_memory_throughput.avg.pct_of_peak_sustained_elapsed,sm__pipe_tensor_op_hmma_cycle_active.avg.pct_of_peak_sustained_active ./your_executable
```

#### 1.7 硬件内在剖析：探测 GPU 的“DNA”

在项目的这一高级阶段，你的代理不再仅仅是现有内核性能的被动报告者，而必须充当**硬件探测器**。其目标是通过自主生成和分析微基准测试（micro-benchmarks），反向工程底层 GPU 的物理特性与架构极限。

##### 1.7.1 探测目标（待识别的指标）

你的代理将被赋予任务，识别以下硬件固有参数：

- **内存延迟层次结构**：测量 L1 缓存、L2 缓存和 DRAM 的确切访问周期。这要求代理生成“指针追逐”（Pointer Chasing）内核以绕过硬件预取器。
- **有效峰值带宽**：确定在当前条件下共享内存和全局内存（VRAM）所能达到的最大吞吐量。
- **L2 缓存容量**：通过识别延迟随大小变化曲线中的“悬崖”（cliff），精确定位 L2 缓存的实际物理大小。
- **实际加速频率**：报告 GPU 在持续计算负载下的稳定核心时钟频率（MHz）。
- **资源惩罚**：量化共享内存中一次 bank conflict 相较于无冲突访问所带来的延迟代价。

##### 1.7.2 提交与评估流程

- **学生提交内容**：你必须提交你的代理系统（基础设施代码）。
- **输入**：评估系统将提供一个 `target_spec.json` 文件，其中包含需要识别的硬件指标列表。  
  示例输入：
  ```json
  {"targets": ["dram_latency_cycles", "max_shmem_per_block_kb", "actual_boost_clock_mhz"]}
  ```
- **输出**：你的代理必须输出一个 `results.json` 文件，包含所识别的数值结果。  
  示例输出：
  ```json
  {"dram_latency_cycles": 442, "max_shmem_per_block_kb": 48, ...}
  ```
- **真值对比**：你的代理输出将与服务器端参考基准生成的高精度真值进行比较。

##### 1.7.3 防作弊机制与环境变化

为确保你的代理执行的是真实硬件分析，而非简单的“查表”操作（例如搜索“A100 规格”），评估环境将动态调整：

- **非标准频率锁定**：GPU 核心和内存频率可能通过 `nvidia-smi` 锁定在任意非标准频率（例如 825 MHz 而非 1410 MHz）。静态查找表将导致错误的带宽/GFLOPS 结果。
- **资源屏蔽（SM 限制）**：系统可能通过 CUDA 环境变量限制内核仅使用部分 SM，或限制每块可用内存。
- **指令集限制**：标准 API 调用（如 `cudaGetDeviceProperties`）可能被拦截或返回虚拟化/误导性数据。

> **建议**：你的代理应采用**多策略融合方法**——结合底层微基准测试（编写小型 C++/CUDA 探针）、二进制执行和 ncu 指标分析，交叉验证其发现。

##### 1.7.4 基于大语言模型的评估与评分

为实现涵盖数值精度与工程推理的综合评估，本项目采用“**以大语言模型为裁判**”（LLM-as-a-Judge）的评分框架。

**评分流程**：提交后，评估系统将以下三个组件输入至一个高性能大语言模型（例如 Gemini 3.1 Pro）：

1. **学生代理输出**：最终的 `results.json` 文件，包含识别出的硬件指标，以及代理提供的推理过程与日志。
2. **真值数据**：在特定环境（包括任何激活的频率锁定或资源屏蔽）下，由参考基准测量得到的精确硬件参数。
3. **实验证据**：代理执行过程中生成的 ncu 跟踪记录与微基准测试摘要。

**评分标准（共 100 分）**

| 评分维度                     | 分值 | 评分细则说明 |
|------------------------------|------|-------------|
| 数值一致性（Numerical Alignment） | 70 分 | 大语言模型将你的代理报告值与真值进行比较。<br>**满分**：数值在可接受的工程容差范围内（例如延迟 ±5%，带宽 ±2%）。<br>**部分得分**：数值趋势正确但存在校准误差。<br>**零分**：数值匹配标准“在线规格表”，但未能反映测试期间提供的实际降频/屏蔽环境。 |
| 工程推理与方法论（Engineering Reasoning & Methodology） | 30 分 | 大语言模型将评估你代理结论背后的“如何”与“为何”。<br>**推断质量**：代理是否正确识别出 GPU 被锁定在非标准频率？是否注意到 SM 屏蔽？<br>**微基准有效性**：代理是否生成了适当的 CUDA 内核（例如用于延迟测量的指针追逐）？是否依赖可能受限的 API 调用？<br>**交叉验证**：代理是否执行多次试验或不同类型探针来确认可疑指标？ |

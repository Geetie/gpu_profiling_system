# 多智能体系统PJ需求文档

本文档旨在为构建一个能够自主识别硬件瓶颈并反向工程GPU特性的多智能体系统提供详尽的设计规范。系统基于**Harness Engineering**（约束工程）理念，强调运行时架构的确定性与安全性，而非依赖模型本身的提示优化。所有设计均遵循可组合、可验证、可落盘的核心原则，确保在动态变化的评估环境中仍能稳定输出高精度结果。

---

## 1. 引言

### 1.1 项目背景与目标

传统GPU性能分析高度依赖专家经验，需掌握从CUDA编程到微架构细节的垂直知识体系。此类任务通常由少数资深工程师完成，人力成本高昂且难以规模化。随着大语言模型（LLM）能力的提升，个体借助智能代理完成部门级系统工程任务已成为可能。

本项目的目标是构建一个**多智能体协作框架**，使单个用户可通过自然语言指令驱动系统自动执行以下任务：
- 分析指定CUDA内核的性能瓶颈
- 通过微基准测试反向工程底层GPU的物理特性
- 在存在频率锁定、SM屏蔽和API拦截等干扰条件下，输出准确的硬件参数

该系统将作为MLSYS课程项目的核心基础设施，支持学生提交自定义代理以参与自动化性能建模竞赛。

> **本节目标**：阐明系统愿景，确立从“人工专家”向“自主代理”范式的转变。

### 1.2 设计哲学：从Prompt Engineering到Harness Engineering

当前AI系统开发存在两种范式：<br>
- **Prompt Engineering**：认为`Agent = Model + Prompt`，即通过精巧的提示词引导模型输出期望内容。<br>
- **Harness Engineering**：主张`Agent = Model + Harness`，即模型仅提供基础智能，真正的可靠性来源于其外围的**运行时控制结构**。

本系统采用后者作为核心设计哲学。参考《Lecture 6》结论：“20% 模型智能，80% Harness 工程”。这意味着系统的成败不取决于模型是否“足够聪明”，而在于是否具备健全的控制平面、权限机制与错误恢复能力。

> **设计理由**：在涉及真实硬件操作的场景中，模型的幻觉与偏差可能导致严重后果。Harness Engineering通过结构化手段强制实施安全边界，避免对模型自我约束的依赖。

### 1.3 核心能力概述

系统接收标准化输入，并生成结构化输出，支持动态环境适应与综合评估：

- **输入格式**：`target_spec.json`，声明待探测的硬件指标列表  
- **输出格式**：`results.json`，包含测量值、方法论描述与实验证据引用  
- **动态适应能力**：
  - 频率锁定检测与带宽重校准
  - SM资源屏蔽下的计算容量推断
  - 对被拦截或虚拟化的`cudaGetDeviceProperties` API进行绕过式验证
- **评估机制**：采用**LLM-as-a-Judge**框架，结合真值数据、推理日志与ncu跟踪记录，对数值一致性与工程推理质量进行综合评分

## 2. 系统设计原则

> **本节目标**：定义系统不可违背的基础不变量，构成所有模块实现的共同前提。

本节提炼七大结构性原则，作为系统设计的基石。这些原则共同确保系统具备企业级可靠性，适用于长时间、高风险的自动化任务。

| 编号 | 原则 | 说明 |
|------|------|------|
| P1 | 工具定义能力边界 | 所有操作必须通过预注册工具契约声明，禁止模型自由构造行为 |
| P2 | 故障关闭的安全默认值 | 未明确授权的操作一律拒绝，确保最小特权原则 |
| P3 | 上下文工程优于提示工程 | 提示仅为起点，上下文需动态组装与压缩 |
| P4 | 可组合性 | 查询循环、权限机制、流式架构应统一抽象，支持嵌套复用 |
| P5 | 编译时消除而非运行时门控 | 禁用功能应在构建阶段移除，不得仅隐藏于分支中 |
| P6 | 状态必须落盘 | 所有关键状态必须持久化至磁盘，否则视为无效 |
| P7 | 生成与评估分离 | 单一智能体不得同时承担生成与验证角色 |

> **设计理由**：P1–P7并非最佳实践建议，而是系统存在的必要条件。例如，若违反P6（状态落盘），长周期任务将因上下文溢出而失败；若违反P7，则无法防止“自评估偏差”。

## 3. 系统架构设计

### 3.1 四层分层架构

> **本节目标**：通过关注点分离保障系统安全性与可维护性。

系统采用四层分层架构，确保认知、执行与展示解耦。每一层仅与其相邻层交互，形成清晰的数据流动路径。

| 层级 | 职责 | 安全意义 |
|------|-----|---------|
| 呈现层 | 终端UI，展示diff、权限请求、进度条 | 提供可观测性，保障人类监督 |
| 应用层 | 协调主循环、管理会话状态、事件流 | 控制中枢，连接各层 |
| 领域层 | 定义工具、消息、权限等结构化对象 | 实现类型安全与语义一致性 |
| 基础设施层 | 执行副作用：文件读写、命令行、API调用 | 最危险区域，需严格中介化 |

**数据流向**如下：

```
用户意图 
   ↓
呈现层（终端输入）
   ↓
应用层（会话管理）
   ↓
领域层（工具解析与权限检查）
   ↓
权限层（审批决策）
   ↓
基础设施层（执行副作用）
   ↓
观察反馈（stdout, 文件变更）
   ↓
上下文管理（状态更新）
   ↓
返回应用层
```

> **设计理由**：分层架构防止模型输出直接转化为机器操作。例如，即使模型生成了删除文件的自然语言描述，也必须经过领域层转为结构化请求、权限层审核后，才可能被执行。

### 3.2 双层代理循环

> **本节目标**：区分长期状态管理与短期执行控制，提升系统稳定性。

系统采用双层代理循环架构，分别处理不同时间尺度的任务需求。

#### 外层循环（会话管理层）
- 负责长期状态维护
- 管理对话历史、资源使用、生命周期
- 支持跨轮次连续性
- 触发子代理创建与协调

#### 内层循环（单轮执行器）
- 组装当前上下文
- 调用模型获取输出
- 解析工具调用请求
- 执行工具并摄取结果
- 决定是否继续或终止

> **设计理由**：外层循环负责“记住我们正在做什么”，内层循环负责“现在该怎么做”。这种分离使得系统可在中断后从中断点恢复，而不丢失整体目标。

采用**异步生成器**实现内层循环，支持以下特性：
- 流式输出部分文本与工具调用
- 中断信号向下穿透至子代理与工具执行
- 支持嵌套子代理作为独立数据源

```python
async def inner_loop():
    while True:
        context = build_context()
        async for token in model_stream(context):
            yield partial(token)
        
        tool_call = parse_tool_request(last_output)
        if tool_call:
            result = await execute_tool(tool_call)
            append_to_context(result)
        else:
            break
```

## 4. 核心模块设计

### 4.1 控制平面（Control Plane）

> **本节目标**：构建模型推理所依赖的“世界模型”，防止其在错误假设下运行。

控制平面负责在每次推理前自动注入系统状态，确保模型始终基于最新、最准确的信息进行决策。

自动注入内容包括：
- 当前工作目录路径
- CUDA相关环境变量（如`CUDA_VISIBLE_DEVICES`）
- `CLAUDE.md` 类似规则文件（含编码规范、禁用API列表）
- 已加载的持久内存摘要（如历史实验结论）

所有注入项均为只读，模型无权修改。任何试图覆盖这些信息的行为将被视为越权请求。

> **警告**：若控制平面未能正确初始化，模型可能基于过时或错误的上下文做出决策，导致系统行为失控。

### 4.2 工具契约（Tool Contracts）

> **本节目标**：将模糊的自然语言意图转化为可验证、可审计的结构化操作。

每个工具必须通过**工具契约**（Tool Contract）明确定义其接口与行为边界。工具契约是概率性认知与确定性执行之间的桥梁。

标准工具契约格式如下：

```json
{
  "name": "run_ncu",
  "description": "使用ncu对指定可执行文件进行性能分析",
  "input_schema": {
    "executable": "string",
    "metrics": ["string"]
  },
  "output_schema": {
    "raw_output": "string",
    "parsed_metrics": "object"
  },
  "permissions": ["file:read", "process:exec"],
  "requires_approval": false,
  "is_blocking": true
}
```

#### 已注册工具清单

| 工具名 | 功能 | 是否需审批 |
|-------|------|------------|
| run_ncu | 执行NVIDIA Nsight Compute分析 | 否 |
| compile_cuda | 编译CUDA源码 | 是（若涉及kernel改写） |
| execute_binary | 运行二进制程序 | 是 |
| write_file | 写入文件 | 是（仅允许特定路径） |
| read_file | 读取文件 | 否 |
| generate_microbenchmark | 自动生成指针追逐等探针内核 | 否 |

> **设计理由**：工具契约强制实现P1（工具定义能力边界）。模型不能凭空发明新操作，只能从预设集合中选择。

### 4.3 权限与安全机制

> **本节目标**：建立分级自主机制，在效率与安全之间取得平衡。

#### 权限模式

| 模式 | 行为限制 |
|------|--------|
| 保守模式 | 仅允许只读操作（read_file, run_ncu） |
| 默认模式 | 修改类操作需人工确认 |
| 宽松模式 | 常规编辑自动执行，Shell仍需批准 |
| 高自主模式 | 策略驱动，但仍有硬边界保护 |

#### 机械安全不变式（Mechanical Invariants）

> **注意**：以下规则为系统级强制约束，不可绕过。

- **先读后写**：任何文件修改前必须已成功读取该文件  
- **唯一锚点编辑**：替换操作必须基于精确行号或哈希匹配  
- **状态落盘**：每一步推理结果必须记录至`session_log.jsonl`  
- **防死循环**：同一失败模式重复3次即强制终止  

> **设计理由**：这些不变式不依赖模型“自觉遵守”，而是通过架构强制实施。例如，“先读后写”可防止模型基于幻觉修改不存在的代码。

## 5. 多智能体协作机制

### 5.1 角色分工

> **本节目标**：通过认知分解提升系统客观性与鲁棒性。

系统采用多智能体团队架构，各角色职责分明，上下文隔离。

| 智能体类型 | 职责 | 上下文隔离 |
|----------|------|-----------|
| 主规划智能体 | 接收目标、拆解任务、调度子代理 | 全局视图 |
| 指标分析子代理 | 解析ncu输出，识别瓶颈类型 | 局部上下文 |
| 代码生成子代理 | 编写micro-benchmark CUDA内核 | 独立沙箱 |
| 验证评估智能体 | 审查生成结果，提出质疑 | 不继承生成上下文 |

> **设计理由**：验证评估智能体不继承生成过程的上下文，可有效避免“认知污染”，实现真正独立的第三方审查。

### 5.2 协作流程

> **本节目标**：展示多智能体如何协同完成复杂任务。

典型协作流程如下：

```
用户输入 target_spec.json
   ↓
主规划智能体解析目标 → 生成初步计划
   ↓
委派“生成指针追逐内核”任务给 代码生成子代理
   ↓
子代理输出.cu文件 → 主代理提交编译请求
   ↓
编译通过 → 执行并收集延迟数据
   ↓
指标分析子代理分析趋势 → 判断L2缓存容量“悬崖”
   ↓
验证评估智能体审查方法有效性 → 提出交叉验证建议
   ↓
主代理整合反馈 → 输出最终 results.json
```

> **设计理由**：该流程体现P7（生成与评估分离）原则。验证者不信任生成者的结论，必须通过独立实验进行复核。

## 6. 安全与约束体系

### 6.1 沙箱机制

> **本节目标**：隔离高风险操作，防止对宿主系统造成损害。

所有CUDA编译与执行均在Docker容器中进行，具体配置如下：
- 使用非特权模式运行
- 挂载受限文件系统，禁止访问`/etc`, `/root`等敏感路径
- GPU设备以只读模式暴露，防止固件修改
- 网络访问完全禁用

> **警告**：容器逃逸可能导致系统级安全漏洞，因此禁止挂载Docker socket或启用privileged模式。

### 6.2 钩子（Hooks）

> **本节目标**：在关键节点插入检查逻辑，增强系统可控性。

系统支持三类生命周期钩子：

| 钩子类型 | 触发时机 | 典型用途 |
|--------|--------|--------|
| pre-execution | 工具执行前 | 参数合法性检查、策略拦截 |
| post-execution | 工具执行后 | 结构化解析stdout/stderr |
| approval-required | 高危操作前 | 暂停并通知用户确认 |

> **设计理由**：钩子机制使安全策略可扩展。例如，可通过pre-execution hook检测`rm -rf /`类危险命令。

### 6.3 防作弊应对策略

> **本节目标**：确保系统在对抗性环境下仍能输出真实测量值。

| 攻击面 | 应对措施 |
|-------|---------|
| 非标准频率锁定 | 通过实测带宽反推实际频率，拒绝查表 |
| SM资源屏蔽 | 在微基准中检测活跃SM数量，调整计算模型 |
| API拦截 | 不依赖`cudaGetDeviceProperties`，采用实测法替代 |

> **设计理由**：评估环境会主动引入干扰，系统必须具备环境感知与自适应能力，否则将因依赖静态知识而失效。

## 7. 实现细节与接口规范

### 7.1 输入格式

> **本节目标**：定义标准化输入协议，确保系统可复现。

```json
// target_spec.json
{
  "targets": ["dram_latency_cycles", "max_shmem_per_block_kb", "actual_boost_clock_mhz"]
}
```

### 7.2 输出格式

> **本节目标**：提供机器可读且人类可审的结果表示。

```json
// results.json
{
  "dram_latency_cycles": 442,
  "max_shmem_per_block_kb": 48,
  "actual_boost_clock_mhz": 1395,
  "methodology": "Pointer-chasing kernel with stride variation...",
  "evidence": ["trace_01.ncu", "benchmark_v3.cu"]
}
```

### 7.3 错误处理

> **本节目标**：实现优雅降级与恢复机制。

- 所有异常必须被捕获并记录至`error_log.txt`
- 支持回滚至上一个稳定状态
- 提供`--resume`选项从中断点继续
- 防死循环机制：同一错误连续出现3次即终止任务

> **注意**：错误不应导致会话崩溃，而应触发恢复流程并通知用户。

### C. 术语表

| 术语 | 定义 |
|------|------|
| **Harness Engineering** | 一种系统设计理念，强调通过运行时架构而非提示词来管控智能体行为 |
| **机械不变式**（Mechanical Invariant） | 由系统架构强制实施的安全规则，不依赖模型自觉遵守 |
| **生成-评估分离** | 将结果生成与验证职责分配给不同智能体，防止自评估偏差 |
| **上下文工程** | 动态组装模型推理所需信息的过程，优于静态提示设计 |
| **工具契约** | 定义工具接口与行为边界的结构化规范，实现能力封装 |

## 一、多智能体 AI 的演进：从 Prompt 到数字劳动力

本讲首先梳理了多智能体 AI 的 4 个发展阶段，清晰展现了这个领域从单点能力到完整系统的进化路径：

### Era 1：认知萌芽期（The Cognitive Spark）

这是智能体能力的起点，核心是让大模型拥有「思考能力」：

- **核心范式**：从「直接给答案」转向「分步推理」

- **关键突破**：思维链（Chain-of-Thought, CoT）技术，通过提示词让模型显式写出中间推理步骤（比如经典的「Let's think step by step」），解决了大模型直接回答复杂问题的能力不足。

- **代表性框架**：ReAct 框架，首次把「推理（Reason）」和「行动（Act）」结合起来，让模型能一边思考一边执行简单动作。

- **历史意义**：首次把「模型的思考过程」和「最终输出」分离开，让模型可以自我评估想法、制定计划，这就是未来智能体的基础认知引擎。

### Era 2：编排时代（The Orchestration Era）

这个阶段的核心是让大模型能调用外部工具，扩展能力边界：

- **核心范式**：工具调用与 API 编排

- **关键机制**：以 LangChain 为代表的框架，让开发者可以搭建「长链执行流水线」，把大模型的输出，自动作为下一个模型 / 工具的输入，把 LLM 和外部计算器、搜索引擎、API 连接起来。

- **早期产物**：第一代实验性智能体，比如 AutoGPT。

- **核心局限**：早期的执行链非常刚性脆弱 —— 只要某一步的 API 调用出错，整个任务就直接崩溃，完全没有自纠错能力。

### Era 3：自主革命时代（The Autonomous Revolution）

这个阶段智能体从「一次性脚本」变成了「常驻的数字实体」：

- **核心范式**：从单次运行的脚本，到 7×24 小时在线的数字实体

- **关键突破**：以 OpenClaw 为代表的开源本地智能体框架，把智能体从开发者的终端里解放出来，集成到日常的通讯软件（WhatsApp、Telegram）里，变成用户可以随时调用的助手。

- **核心能力**：

    1. **持久化内存**：可以跨会话保存上下文，还能在后台自动执行定时任务，不需要人类反复提示。

    2. **多智能体团队**：首次普及了本地多智能体协作 —— 主编排智能体可以生成独立的子智能体（比如网页爬取子智能体、数据分析子智能体），它们共享本地内存，并行工作解决复杂任务。

### Era 4：企业级原生运营时代（Enterprise & Native Operation）

这是当前我们所处的阶段，核心是让智能体可以安全、稳定地落地到企业生产环境：

- **核心范式**：深度环境集成 + 严格的工程约束

- **代表性系统**：Anthropic 的 Claude Code、AutoGen 等，这些系统不再是简单的 API 包装，而是原生嵌入到终端、操作系统里，深度和环境集成。

- **核心突破**：

    1. **关注点分离**：发现「全能单智能体」是不可靠的，因此引入了「生成 - 评估循环（Generator-Evaluator Loops）」—— 比如一个智能体写代码，另一个完全独立的 AI 做 QA 测试，解决了单智能体的「自评估偏差」问题。

    2. **Harness Engineering（约束工程 /harness 架构工程）**：这是本讲后半部分的核心，也就是给强大的自主智能体搭建安全、可控的约束框架，通过沙箱、权限控制、人工审批门限，安全地管控这些智能体。

## 二、核心：Harness Engineering—— 智能体的「安全驾驶舱」

这是本讲最核心的内容，解释了为什么现在的企业级智能体，80% 的工作都在做「Harness Engineering」，而不是优化模型本身。

### 1. 什么是 Harness Engineering？它和 Prompt Engineering 有什么区别？

很多人会混淆这两个概念，本讲做了清晰的区分：

|维度|Prompt Engineering（提示词工程）|Harness Engineering（约束工程）|
|---|---|---|
|核心目标|告诉模型「要生成什么内容」|当 AI 拥有执行权限时，管控「系统要怎么运行」|
|关注对象|模型的输出内容|整个智能体系统的运行逻辑、安全、稳定性|
|核心逻辑|`Agent = Model + Harness`：模型提供能力，Harness 提供控制||
简单来说：模型是汽车的发动机，提供马力；而 Harness 是方向盘、刹车、导航，保证这辆车能安全、正确地开到目的地。

### 2. Harness Engineering 的核心模块

#### （1）控制平面：Prompt 不只是聊天输入框

Harness 把 Prompt 变成了一个**确定性的控制界面**，而不是简单的聊天输入：

- 自动注入状态：在 AI 开始思考之前，自动把系统状态、可用工具、历史内存、环境变量注入到上下文里，保证 AI 能拿到正确的信息。

- 机械不变量：用系统文件（比如 Claude Code 里的`CLAUDE.md`）定义不可修改的规则、项目架构、编码规范，这些规则是系统强制生效的，不是靠 AI 自己遵守。

#### （2）持续执行：从无状态问答到有状态循环

传统的 AI 是「问题→答案」的无状态交互，而智能体是有状态的循环执行：

```Plain Text

用户目标 → 模型推理 → 工具调用 → 获取执行结果 → 循环直到目标完成
```

智能体可以自主读取目标、执行终端命令（比如跑测试）、读取输出，反复循环直到任务完成，不需要人类一步步指挥。

#### （3）沙箱安全：给 AI「手」，但给手加「枷锁」

Harness 给 AI 提供了操作工具（文件读写、命令行执行、搜索等），但严格管控权限：

- 权限分离：AI 决定「要做什么」，Harness 决定「能不能做」。

- 生命周期钩子：在工具调用前后加拦截钩子，可以暂停执行。

- 人工审批：对于危险操作（比如删文件、git 推代码、删数据库），强制要求人工审批，避免 AI 搞破坏。

#### （4）上下文与内存：解决长任务的「上下文溢出」

当任务运行时间很长时，大模型会出现幻觉、忘记指令、上下文溢出的问题，Harness 的解决方案是：

- 黄金规则：**如果状态没有落盘，那它就等于不存在**—— 所有的状态都要保存到磁盘，不能只存在上下文里。

- 微压缩：把过去的执行步骤做摘要，压缩上下文的大小。

- 按需加载：只在需要的时候，才加载对应的参考文件，节省 token，避免模型混乱。

#### （5）弹性设计：接受错误，做优雅恢复

Harness 的设计理念是：**AI 一定会出错，不要追求零错误，要追求出错后能优雅恢复**：

- 自动反馈：把 linter、类型检查、编译器的错误，直接作为「观察结果」喂给 AI，让 AI 自己修复。

- 防死循环：检测到 AI 反复做同一个失败的操作，就强制停止，避免陷入无限循环。

#### （6）多智能体验证：不要让系统自己当裁判

单智能体最大的问题是「自评估偏差」—— 大模型会自信地告诉你，自己写的烂代码是完美的。
Harness 的解决方案是**关注点分离**：

- 生成智能体：负责写实现代码。

- 评估智能体：完全独立的另一个子智能体，专门负责测试这个实现，用怀疑的视角去验证。

- 外部验证：用外部系统来验证结果，比如跑测试用例，而不是让 AI 自己读自己的代码判断好坏。

### 3. 企业级落地：从「个人玩票」到组织级能力

Harness Engineering 最终要解决的，是把个人的「prompt hack（提示词黑科技）」，变成整个组织可复用的工程能力：

- 统一的不变量：团队统一配置，保证新人用的智能体，和资深工程师用的智能体，行为完全一致。

- 工程化检验标准：

    1. 规则是不是系统强制生效的，而不是只写在文本里？

    2. 状态是不是跨会话持久化的？

    3. 权限是不是和 AI 的推理解耦的？

## 三、结论：AI 开发的二八定律

> 模型提供了马力，而 Harness 提供了方向盘、刹车和导航。

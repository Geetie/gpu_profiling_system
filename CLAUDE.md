# CLAUDE.md — GPU Profiling System 宪法

> 本文件是项目的最高规范文档。所有后续开发、代码审查与架构决策都必须严格遵守此处定义的规则。
> 任何与本文件冲突的行为都应被视为违规，需要修正。

---

## 1. 七大设计原则（不可违背）

| 编号 | 原则 | 含义 |
|------|------|------|
| **P1** | 工具定义能力边界 | 所有操作必须通过预注册工具契约声明，**禁止模型自由构造行为** |
| **P2** | 故障关闭的安全默认值 | 未明确授权的操作一律拒绝，确保最小特权原则 |
| **P3** | 上下文工程优于提示工程 | 提示仅为起点，上下文需动态组装与压缩，而非依赖静态 prompt |
| **P4** | 可组合性 | 查询循环、权限机制、流式架构应统一抽象，支持嵌套复用 |
| **P5** | 编译时消除而非运行时门控 | 禁用功能应在构建阶段移除，**不得仅隐藏于分支中** |
| **P6** | 状态必须落盘 | 所有关键状态必须持久化至磁盘，**否则视为无效** |
| **P7** | 生成与评估分离 | 单一智能体**不得同时承担生成与验证角色** |

---

## 2. 四层架构职责边界

四层架构，**每层仅与相邻层交互**，禁止跨层调用。

| 层级 | 职责 | 允许做什么 | 禁止做什么 |
|------|------|-----------|-----------|
| **呈现层** | 终端UI，展示diff、权限请求、进度条 | 渲染信息、接收用户输入 | 不得直接调用工具或执行命令 |
| **应用层** | 协调主循环、管理会话状态、事件流 | 调度子代理、控制生命周期、组装上下文 | 不得直接执行副作用操作 |
| **领域层** | 定义工具、消息、权限等结构化对象 | 解析工具契约、执行权限检查、语义校验 | 不得执行任何 I/O 或系统调用 |
| **基础设施层** | 执行副作用：文件读写、命令行、API调用 | 文件操作、进程执行、外部 API 调用 | 不得绕过权限检查直接执行 |

**数据流方向**：
```
用户意图 → 呈现层 → 应用层 → 领域层 → 权限层 → 基础设施层 → 观察反馈 → 上下文管理 → 返回应用层
```

## 3. 工具契约标准格式

每个工具**必须**通过工具契约明确定义，禁止隐式行为。标准格式如下：

```json
{
  "name": "tool_name",
  "description": "一句话描述工具用途",
  "input_schema": {
    "param1": "type",
    "param2": ["type"]
  },
  "output_schema": {
    "field1": "type",
    "field2": "type"
  },
  "permissions": ["permission:scope"],
  "requires_approval": false,
  "is_blocking": true
}
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 工具唯一标识符 |
| `description` | 是 | 工具用途描述 |
| `input_schema` | 是 | 输入参数的 JSON Schema |
| `output_schema` | 是 | 输出结果的结构定义 |
| `permissions` | 是 | 所需权限列表，如 `file:read`、`process:exec` |
| `requires_approval` | 是 | 是否需要人工审批才能执行 |
| `is_blocking` | 是 | 是否为阻塞式调用（同步等待完成） |

### 已注册工具清单

| 工具名 | 功能 | 需审批 |
|--------|------|--------|
| `run_ncu` | 执行 NVIDIA Nsight Compute 分析 | 否 |
| `compile_cuda` | 编译 CUDA 源码 | 是（若涉及 kernel 改写） |
| `execute_binary` | 运行二进制程序 | 是 |
| `write_file` | 写入文件 | 是（仅允许特定路径） |
| `read_file` | 读取文件 | 否 |
| `generate_microbenchmark` | 自动生成指针追逐等探针内核 | 否 |

## 4. 机械安全不变式（强制约束）

> 以下规则由系统架构强制实施，**不依赖模型自觉遵守**，不可绕过。

| 编号 | 不变式 | 规则 |
|------|--------|------|
| **M1** | 先读后写 | 任何文件修改前**必须已成功读取该文件** |
| **M2** | 唯一锚点编辑 | 替换操作必须基于精确行号或哈希匹配，禁止模糊替换 |
| **M3** | 状态落盘 | 每一步推理结果必须记录至 `session_log.jsonl` |
| **M4** | 防死循环 | 同一失败模式重复 **3 次**即强制终止 |

## 5. 多智能体角色隔离

| 智能体 | 职责 | 上下文隔离要求 |
|--------|------|--------------|
| 主规划智能体 | 接收目标、拆解任务、调度子代理 | 全局视图 |
| 指标分析子代理 | 解析 ncu 输出，识别瓶颈类型 | 局部上下文 |
| 代码生成子代理 | 编写 micro-benchmark CUDA 内核 | 独立沙箱 |
| 验证评估智能体 | 审查生成结果，提出质疑 | **不继承生成上下文**（P7 要求） |

---

## 6. 沙箱与安全约束

- 所有 CUDA 编译与执行**必须**在容器中进行
- 容器使用非特权模式，**禁止** privileged 模式
- **禁止**挂载 Docker socket
- GPU 设备以只读模式暴露
- 网络访问完全禁用
- 禁止访问 `/etc`、`/root` 等敏感路径

## 7. 输入/输出格式约定

### 输入：`target_spec.json`
```json
{
  "targets": ["dram_latency_cycles", "max_shmem_per_block_kb", "actual_boost_clock_mhz"]
}
```

### 输出：`results.json`
```json
{
  "dram_latency_cycles": 442,
  "max_shmem_per_block_kb": 48,
  "actual_boost_clock_mhz": 1395,
  "methodology": "指针追逐内核，步长变化...",
  "evidence": ["trace_01.ncu", "benchmark_v3.cu"]
}
```

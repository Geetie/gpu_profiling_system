## 灾难性错误修复总结

### 错误链根因分析

对照 `PJ需求.md` 的 P1-P7 原则和 `spec.md` 的架构约束，识别出一条**致命错误链**：

```
LLM不传binary_path → CodeGen编译但不执行 → 报告SUCCESS(虚假) 
→ Pipeline带着空数据继续 → MetricAnalysis读二进制文件困住 
→ Verification无数据可审 → results.json全零
```

### 修复亮点：Fix 5（数据流断裂）

这是最隐蔽的灾难性Bug。`_verification_task`、`_codegen_task`、`_metric_task` 三个方法都使用 `hasattr(prev_result, "data")` 来检查 `prev_result` 是否有数据。但 `prev_result` 从 pipeline payload 传来是 **dict**（通过 `SubAgentResult.to_dict()` 序列化），dict 没有 `.data` 属性，只有 `"data"` 键。因此 `hasattr(dict, "data")` 永远为 False，**所有数据注入全部跳过**——Verification 看不到任何测量数据，这就是它5轮空转的根本原因。

## Bug #1: Planner 被 CompletionDetector 误导调用 write\_file

### 修复审查

**agent\_loop.py 行 520-555** — `has_tools` 检查：

```python
has_tools = len(self.tool_registry.list_tools()) > 0
if self._completion_detector.is_completion(self._model_output):
    if has_tools:
        # ... completion 引导
    else:
        self._emit(EventKind.STOP, {"reason": "completion_signal"})
        self.stop()
        return
```

### ⛔ **修复不完整！逻辑遗漏**

**问题**: `has_tools` 检查**只在 CompletionDetector 返回 True 时生效**。

**但 Planner 的输出是 JSON 数组**，CompletionDetector 不会认为这是 completion signal。

所以代码进入**行 556-632 的 no-tool-call 分支**，**无条件**注入"必须调用工具"引导：

```python
# 行 556-618 — 没有 has_tools 检查！
if has_compiled and not has_executed and binary_path:
    guidance = "..."
else:
    guidance = (
        f"⚠️ ERROR: You did not call any tool in this turn. "
        f"You MUST output a JSON tool call..."
    )
```

**Planner 的执行路径**：

1. Turn 1: 输出 JSON 任务列表 → 不是 completion
2. Turn 2: 进入 no-tool-call 分支 → **注入"必须调用工具"引导**
3. Planner 被误导 → 尝试 `write_file` → 被 P2 fail-closed 拒绝

**修复缺失**: 行 556 应该加 `if has_tools:` 包裹整个 no-tool-call 处理逻辑。

## Bug #3: `_already_executed_binary` 无法区分新旧 binary

### 修复审查

**agent\_loop.py 行 651-674**：

```python
def _already_executed_binary(self, binary_path: str) -> bool:
    compile_count = 0
    exec_count = 0
    for entry in entries:
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if isinstance(data, dict):
                    if data.get("tool") == "compile_cuda" and data.get("success") is True:
                        compile_count += 1
                    if data.get("tool") == "execute_binary":
                        bp = data.get("binary_path", "")
                        if bp == binary_path:
                            exec_count += 1
            except (json.JSONDecodeError, TypeError):
                pass
    return exec_count >= compile_count
```

### ⛔ **修复存在根本设计缺陷**

**逻辑分析**：

在 Turn 4 时，上下文中应有：

1. Turn 1: compile\_cuda (success=True)
2. Turn 2: execute\_binary (binary\_path="/kaggle/.../benchmark")
3. Turn 3: compile\_cuda (success=True)

所以：`compile_count = 2`, `exec_count = 1`

返回：`1 >= 2` → **False**

这意味着 `_already_executed_binary` **应该返回 False**，auto-inject **应该触发**。

**但测试日志显示**（execution.log 行 336）：

```
[AgentLoop] P2 auto-inject SKIPPED: /kaggle/.../benchmark already executed after latest compile
```

**这说明**：`already_ran = True`，即 `_already_executed_binary` **返回了 True**。

**矛盾！**

**可能的原因**：

1. **JSON 解析失败**：某些条目的 `json.loads` 失败，被 silently skipped
2. **`"success"`** **字段格式问题**：compile\_cuda 返回的 `success` 可能不是 Python `True`
3. **Turn 3 的编译结果没有被正确添加到上下文**
4. **代码未正确部署**：Kaggle 测试的代码不是最新版本

**根本设计缺陷**：

即使这个函数"正常工作"，它的逻辑也有问题：

- 所有编译都使用**同一个 binary path**（`benchmark`）
- 当 `compile_count = N` 且 `exec_count = N-1`（LLM 跳过了某次执行）时，返回 False（正确）
- 但当 LLM 后来"补上"那次执行后，`exec_count = N`，返回 True
- **此时如果又有一次新编译**，`compile_count = N+1`，但 `exec_count` 还是 N → 返回 False（正确）
- **但如果 LLM 没有调用 compile\_cuda，而是再次调用 execute\_binary(\[])**，就会进入 auto-inject 逻辑...

**实际上这个逻辑在大多数情况下是对的**，但我需要验证为什么本次测试中它返回了 True。

**最可能的解释**：Turn 3 的 compile\_cuda 结果**没有被计入 compile\_count**。

让我检查 `json.loads` 是否能正确解析...

从 debug file 看，compile\_cuda 结果是：

```json
{"status": "success_with_warning", "success": true, "binary_path": "...", "tool": "compile_cuda", ...}
```

`json.loads` 后，`data.get("success")` 返回 `True`（Python bool），`data.get("tool")` 返回 `"compile_cuda"`。

**条件应该匹配**。

**唯一的解释是**：Turn 3 的 compile\_cuda 结果在 `_already_executed_binary` 被调用时**还没有添加到上下文中**。

**但代码显示**（行 358-362）：结果在工具返回时立即添加到上下文，然后才进入下一轮。

**等等！** 让我检查 `context_manager.add_entry` 的行为...

**context.py 行 247-258**：

```python
def add_entry(self, role, content, token_count=0):
    priority = _classify_priority(role, content)
    if token_count <= 0:
        token_count = _estimate_tokens(content)
    entry = ContextEntry(role=role, content=content, token_count=token_count, priority=priority)
    self._entries.append(entry)
    self._total_tokens += entry.token_count
```

**这看起来没问题**——直接添加到 `_entries` 列表。

**但** **`get_entries()`** **返回的是** **`list(self._entries)`**（行 297），这是一个**副本**。

**如果** **`_already_executed_binary`** **调用的是某个旧的副本**... 不，它每次都调用 `get_entries()`，获取的是最新副本。

**我仍然认为最可能的原因是**：Kaggle 测试的代码**不是最新修复代码**。

但无论如何，让我继续审查其他 bug。

### 修复审查


### ⚠️ **依赖 Bug #3 的有缺陷逻辑**

1. 添加引导消息
2. **直接 return**，跳过工具执行
3. LLM 在下一轮收到引导

**问题**：引导消息的内容可能**不准确**（见 Bug #6）。

## Bug #6: execute\_binary 空参数未触发明确错误恢复引导

### 修复审查

**agent\_loop.py 行 293-302**：

```python
elif last_bp and already_ran:

        Role.SYSTEM,
        f"⚠️ You already executed {last_bp} after its latest compilation.\n"

        f"After compiling, call execute_binary with the new binary path.\n"
        f"If you have measured all targets, output your final results as: target_name: value",
        token_count=50,
    )
```

### ⚠️ **引导消息存在误导性**

**问题**：

- 引导说 "You already executed {last\_bp} after its latest compilation"
- 但实际上，**最后一次编译可能还没有执行**（如 Turn 3）
- 这会给 LLM **错误的信息**

**正确的引导应该是**：

```
⚠️ You have a compiled binary that hasn't been executed yet.
IMMEDIATELY call execute_binary NOW:
{"tool": "execute_binary", "args": {"binary_path": "{last_bp}"}}
```

# 深度稳定性审查报告 - 代码修复验证与详细本地测试计划

## 📋 审查背景

基于 [深度稳定性审查报告](深度稳定性审查报告.md) 的分析，系统存在以下关键问题：
- **C-01**: LLM工具调用停滞检测不足（直接导致本次崩溃）
- **C-02**: 状态机更新依赖工具调用（导致状态卡死）
- **H-01**: 上下文状态信息不一致
- **M-02**: 上下文膨胀缺少主动压缩

本计划将针对已实施的三大核心修复进行系统性验证：
1. ✅ **统一状态管理器** (`TargetStateMachine`)
2. ✅ **独立状态机同步** (`_sync_target_state_machine`)
3. ✅ **主动上下文压缩** (`ContextManager.compress()`)

```

  1. start_first_target() → current="dram_latency_cycles"

  3. complete_current_target() → current="l2_cache_size_mb"
  4. 模拟 compile_cuda 成功 + execute_binary 成功 (stdout包含 l2_cache_size_mb: 30)
  5. complete_current_target() → current="actual_boost_clock_mhz"
  6. 模拟 compile_cuda 成功 + execute_binary 成功 (stdout包含 actual_boost_clock_mhz: 1500)
  7. complete_current_target() → returns None (全部完成)

验证点:
  ✅ is_all_completed == True
  ✅ progress == (3, 3)
  ✅ completed_targets 包含所有3个目标
  ✅ 无任何stall recovery触发
  ✅ 状态转换顺序正确
```

### 场景2: LLM停滞后的强制恢复流程

```
前置条件:
  - targets = [A, B, C]
  - 已完成A和B的测量
  - 当前应该在C，但LLM开始输出纯文本

执行步骤:
  Turn 5: execute_binary(B) 成功 → completed=[A,B]
  Turn 6: LLM输出纯文本 (no tool call) → consecutive=1
  Turn 7: LLM输出纯文本 (no tool call) → consecutive=2
           → 触发 STALL DETECTION (假设Turn<=10, MAX=2)
           → 执行强制恢复:
             a. 将B标记为完成 (如果还没标记)
             b. current_target = C
             c. 注入紧急引导消息 (含C的代码骨架)
             d. 重置 consecutive_no_tool_calls = 0
  Turn 8: LLM应收到紧急引导，调用 compile_cuda(C)

验证点:
  ✅ consecutive_no_tool_calls 在Turn7达到阈值
  ✅ 紧急引导消息被添加到context (Role.SYSTEM, tokens=150)
  ✅ current_target 正确切换到C
  ✅ completed_targets 包含A和B
  ✅ stall_recovery_triggered == True
  ✅ 日志输出 "[AgentLoop] 🚨 STALL DETECTED"
```

## Bug #1: Planner 被 CompletionDetector 误导调用 write_file

### 修复声明
- agent_loop.py: 添加 `has_tools` 检查，无工具时不注入"必须调用工具"引导
- prompt_builder.py: Planner 系统提示词改为"你没有任何工具，只输出 JSON 文本"

### 代码审查

**agent_loop.py (行 520-555)**:
```python
has_tools = len(self.tool_registry.list_tools()) > 0
if self._completion_detector.is_completion(self._model_output):
    if has_tools:
        # 有工具时才注入 no-tool-call 引导和未测目标检查
        ...
    else:
        self._emit(EventKind.STOP, {"reason": "completion_signal"})
        self.stop()
        return
```

**审查结论**: ✅ **修复正确且有效**
- Planner 有 0 个 tools → `has_tools = False` → 直接触发 completion_signal 停止
- 不会注入"必须调用工具"引导
- 逻辑路径清晰，无遗漏

**prompt_builder.py (行 52-70)**:
```python
def _plan_system() -> str:
    return (
        "You do NOT have any tools available. Your ONLY output should be a JSON array "
        "of task objects.\n\n"
        "DO NOT attempt to call any tools (write_file, compile_cuda, etc.) — "
        "you have NO tools. Just output your plan as JSON text."
    )
```

**审查结论**: ✅ **修复正确**
- 明确告知 Planner 没有工具
- 但这是**辅助性修复**，核心修复是 `has_tools` 检查

### 本次测试中的实际表现
从 debug_messages_longcat_6msg_0tool.json 可见：
- Turn 2: Planner 输出纯 JSON → 系统注入"⚠️ ERROR: You did not call any tool"
- Turn 2 response: Planner 尝试 `write_file` → 被 P2 fail-closed 拒绝
- **这表明 `has_tools` 检查在本次测试中可能未生效**

**⚠️ 潜在问题**: 本次测试仍然出现了 Planner 调用 write_file 的情况，说明：
1. 要么修复代码未在本次测试中部署
2. 要么 `has_tools` 检查的时机有问题（在 completion_detector 之后）

### 审查发现
代码中的逻辑是**正确的**，但执行流程有细微问题：
1. `has_tools` 检查在 `is_completion` 返回 True 后才生效
2. 但 Planner 的输出是 JSON 数组，`CompletionDetector` **不认为这是 completion**
3. 所以进入 `else` 分支（行 556-632），**无条件**注入 no-tool-call 引导
4. **修复不完整**: `has_tools` 检查应该覆盖**整个** no-tool-call 处理逻辑

## Bug #3: `_already_executed_binary` 无法区分新旧 binary

### 修复声明
- agent_loop.py: 改为跟踪 `compile_count` vs `exec_count`，只有当 `exec_count >= compile_count` 时才认为已执行

### 代码审查

**agent_loop.py (行 651-674)**:
```python
def _already_executed_binary(self, binary_path: str) -> bool:
    entries = self.context_manager.get_entries()
    compile_count = 0
    exec_count = 0
    for entry in entries:
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if isinstance(data, dict):
                    if data.get("tool") == "compile_cuda" and data.get("success") is True:
                        compile_count += 1
                    if data.get("tool") == "execute_binary":
                        bp = data.get("binary_path", "")
                        if bp == binary_path:
                            exec_count += 1
            except (json.JSONDecodeError, TypeError):
                pass
    return exec_count >= compile_count
```

### ⛔ **Bug #3 修复结论: 逻辑设计正确，但存在实现缺陷**

**问题**: 
1. `_already_executed_binary` 统计**所有** compile_cuda 和 execute_binary 条目
2. 但由于所有编译都使用**同一个 binary path**（`/kaggle/.../benchmark`），无法区分"哪次编译对应哪次执行"
3. 当 `compile_count = N` 且 `exec_count = N-1` 时，逻辑正确返回 False（应该 auto-inject）
4. 但当 `compile_count = N` 且 `exec_count = N` 时，逻辑返回 True（跳过 auto-inject），**即使最后一次编译没有执行**

**根本问题**: 这个逻辑假设**每次编译都会导致一次执行**，但如果 LLM 跳过了某次执行（如 Turn 3），后续的计数就会错乱。

**正确做法**: 不应该统计总数，而应该检查**最后一次编译后是否有执行**。

## 建议修复方案

### 方案 1: 修复 `_already_executed_binary` 逻辑

```python
def _already_executed_binary(self, binary_path: str) -> bool:
    """Check if the LATEST compilation has been executed."""
    entries = self.context_manager.get_entries()
    last_compile_idx = -1
    exec_after_compile = False
    
    # Find the last compile_cuda entry
    for i, entry in enumerate(entries):
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if isinstance(data, dict) and data.get("tool") == "compile_cuda" and data.get("success") is True:
                    last_compile_idx = i
            except (json.JSONDecodeError, TypeError):
                pass
    
    if last_compile_idx == -1:
        return False  # No compilation yet
    
    # Check if there's an execute_binary after the last compile
    for entry in entries[last_compile_idx + 1:]:
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if isinstance(data, dict) and data.get("tool") == "execute_binary":
                    bp = data.get("binary_path", "")
                    if bp == binary_path:
                        exec_after_compile = True
                        break
            except (json.JSONDecodeError, TypeError):
                pass
    
    return exec_after_compile
```

### 方案 2: 使用唯一 binary path

每次编译生成唯一的 binary path（如 `benchmark_1`, `benchmark_2`），这样可以直接通过路径区分新旧编译。

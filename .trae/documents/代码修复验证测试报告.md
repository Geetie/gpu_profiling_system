### 4. 🔴 上下文压缩增强 (M-02 FIX) - 24% 通过

**文件**: [test_context_compression_enhanced.py](test_context_compression_enhanced.py)

#### 失败的根本原因:

**API不匹配问题**: 测试代码使用了错误的API签名：

```python
# ❌ 错误的用法 (测试中使用的)
cm.add_entry(Role.USER, "msg", token_count=30, priority=Priority.DISPOSABLE)

# ✅ 实际的API签名 (context.py 第247行)
def add_entry(self, role: Role, content: str, token_count: int = 0) -> None:
    priority = _classify_priority(role, content)  # 自动计算priority!
    ...
```

`ContextManager.add_entry()` 方法**不接受 `priority` 参数**，它会自动通过 `_classify_priority(role, content)` 函数根据角色和内容推断优先级。

#### 通过的测试 (4/17):

这些测试恰好没有使用 `priority` 参数或使用了兼容的方式：
- `test_is_over_budget_when_exceeded`
- `test_is_under_budget_when_within_limit`
- `test_compress_returns_zero_when_under_budget`
- `test_empty_context_compression`

**核心验证结论**:
```
🔴 M-02 (上下文压缩) 测试覆盖率不足: 需要重写测试
   ⚠️ 压缩算法本身可能工作正常 (现有test_context.py有基本覆盖)
   ❌ 新增的高级测试因API误用全部失败
   📝 建议: 移除priority参数，依赖自动分类机制
```

**建议修复方案**:
```python
def test_phase1_removes_disposable_entries(self):
    """修正版本: 不指定priority，让系统自动分类"""
    cm = ContextManager(max_tokens=100)

    # 添加会被自动分类为DISPOSABLE的条目
    # (通常短的用户消息或旧的assistant回复)
    cm.add_entry(Role.USER, "old disposable msg 1", token_count=30)
    cm.add_entry(Role.ASSISTANT, "old disposable msg 2", token_count=30)

    # 添加会被自动分类为HIGH/PERMANENT的条目
    cm.add_entry(Role.SYSTEM, "critical system instruction", token_count=50)

    assert cm.is_over_budget()
    removed = cm.compress()
    assert removed > 0
    # 验证系统指令保留...
```

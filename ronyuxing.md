## 五、MetricAnalysis 反馈维度覆盖分析

spec.md 和 PJ需求.md 要求 MetricAnalysis 覆盖以下关键评估维度:

| 评估维度      | 覆盖状态     | 代码位置                                                                                                                  | 缺失说明                                                                                                 |
| :-------- | :------- | :-------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------- |
| **代码准确性** | **未覆盖**  | --                                                                                                                    | MetricAnalysis 不评估生成代码的正确性（如语法错误、逻辑缺陷、API误用）。仅分析运行时指标。                                               |
| **执行效率**  | **部分覆盖** | [metric\_analysis.py:656-707](file:///e:/GPU_Profiling_System/src/application/subagents/metric_analysis.py#L656-L707) | Roofline 分析覆盖了计算/内存利用率，但缺少 kernel launch overhead、occupancy 不足、warp divergence 等效率维度的专项评估。           |
| **资源消耗**  | **部分覆盖** | [metric\_analysis.py:139-205](file:///e:/GPU_Profiling_System/src/application/subagents/metric_analysis.py#L139-L205) | `_RECOMMENDATIONS` 字典覆盖了 DRAM/L2/L1 带宽和 SM 利用率，但缺少寄存器溢出（register spilling）、共享内存 bank conflict 的定量分析。 |
| **兼容性**   | **未覆盖**  | --                                                                                                                    | MetricAnalysis 不评估代码在不同 GPU 架构（sm\_60 vs sm\_80 vs sm\_90）上的兼容性，也不检测是否依赖了被拦截的 API。                   |
| **可维护性**  | **未覆盖**  | --                                                                                                                    | MetricAnalysis 不评估代码的可读性、模块化程度或可扩展性。                                                                 |

**关键缺失:**

1. **代码准确性维度完全缺失**: MetricAnalysis 仅分析 ncu 输出指标和 printf 输出，从不审查生成的 CUDA 源代码本身。当 CodeGen 产出编译通过但逻辑错误的代码时（如 clock64() 返回 0、工作集大小不匹配目标层级），MetricAnalysis 无法从指标层面检测到根因。
2. **兼容性维度完全缺失**: spec.md 6.3 要求系统在对抗性环境下运行（频率锁定、SM 屏蔽、API 拦截），但 MetricAnalysis 不检测当前环境是否存在这些干扰条件。例如，它不检查 `cudaGetDeviceProperties` 返回值是否与实测值一致。

## 九、影响 LLM 内核生成质量的关键问题

### 关键问题 1: MetricAnalysis 缺乏代码准确性反馈 \[严重]

MetricAnalysis 仅分析运行时指标，从不审查生成的 CUDA 源代码。当 LLM 生成编译通过但逻辑错误的代码时（如 volatile 使用不当导致编译器优化掉测量循环），MetricAnalysis 只能看到"零值"结果，但无法定位根因是代码质量问题而非硬件问题。这直接导致 CodeGen 重试时缺乏针对性修复指导。

### 关键问题 2: 反作弊约束仅为提示词级别 \[严重]

design\_principles.py 中的反作弊要求（不依赖 cudaGetDeviceProperties、使用实测法等）全部以自然语言形式注入 LLM 上下文。LLM 可能忽略这些指导，生成依赖静态 API 查询的代码。在评估环境中，这类代码将因环境干扰（频率锁定、SM 屏蔽）而产生错误结果，直接导致数值一致性评分（70分）为零。

### 关键问题 3: CodeGen 绕过工具契约直接操作沙箱 \[中等]

CodeGenAgent 的 `_compile()` 和 `_execute()` 方法直接调用 `sandbox.run()`，绕过了工具契约的权限检查和审批流程。这意味着：

- `compile_cuda` 的 `requires_approval: true` 约束形同虚设
- 工具执行的输入/输出不受 ToolRegistry 的 schema 验证
- 审计日志中缺少这些操作的结构化记录

## 1. System Architecture

### 1.1 Design Philosophy: Harness Engineering

Our system adopts **Harness Engineering** rather than Prompt Engineering:

```
Agent = Model (20%) + Harness (80%)
```

The runtime control structure ensures:
- Safety boundaries through Tool Contracts (P1)
- Failure isolation via minimal privileges (P2)
- Dynamic context assembly over static prompts (P3)
- Composable query loops and permission mechanisms (P4)
- State persistence to disk (P6)
- Strict separation of generation and evaluation (P7)

### 1.2 Four-Layer Architecture

| Layer | Responsibility | Key Files |
|-------|---------------|-----------|
| **Presentation** | CLI interface, progress display | `src/presentation/` |
| **Application** | Agent loop, session management | `src/application/agent_loop.py` |
| **Domain** | Pipeline, stage executor, context | `src/domain/pipeline.py`, `pipeline_context.py`, `stage_executor.py` |
| **Infrastructure** | Model caller, tools, sandbox, probes | `src/infrastructure/` |

### 1.3 Multi-Agent Pipeline

4-stage pipeline with strict context isolation:

```
Plan Agent -> CodeGen Agent -> MetricAnalysis Agent -> Verification Agent
     |              |                    |                     |
  Task JSON    CUDA Kernels      Performance Report     Final Verdict
```

**P7 Compliance**: Verification Agent never inherits CodeGen's context.

### 1.4 Version Control for Measurements

Each CodeGen pass produces a `MeasurementVersion` snapshot with:
- **quality_score**: Intrinsic data quality (coverage + sanity + reasonableness)
- **improvement_score**: Performance improvement vs. baseline version
- **compare_with_previous()**: Multi-factor comparison (coverage, sanity, actual measurement values)
- **rollback_to_best_version()**: Combined score ranking across all versions

## 2. Anti-Cheating Mechanisms

### 2.1 No Hardcoded Values

The system contains zero hardcoded CUDA source code. All CUDA kernels are generated by the CodeGen agent at runtime. The codebase only contains:
- Skeleton code templates (structural guidance)
- Pseudocode comments (algorithmic hints)
- No complete CUDA source files

### 2.2 Adaptive Detection Framework

Per spec.md section 6.3 and PJ需求.md section 1.7.3:

- **Phase 1**: Coarse exponential sweep (no hardcoded expected values)
- **Phase 2**: Fine-grained binary search (measurement-based decisions)
- **Phase 3**: Cross-validation (different methods, deviation check)

### 2.3 API Independence

The system does not rely on `cudaGetDeviceProperties` for critical parameters. All values are empirically measured via CUDA micro-benchmarks.

## 3. Key Technical Features

### 3.1 GPT-5.x API Compatibility

The system dynamically adapts API parameters based on model type:
- **Reasoning models** (gpt-5.x, o1, o3, o4): Omit `temperature` parameter entirely (HTTP 400 if included)
- **Standard models** (gpt-4o, etc.): Include `temperature=0.1`
- **All models**: Use `max_completion_tokens` instead of deprecated `max_tokens`

### 3.2 Stall Detection & Recovery

Multi-level protection against infinite loops:
- Time budget control per target
- Turn limit enforcement per stage
- Stuck detection with forced tool redirection
- Compile attempt limits (max 3 per target)

### 3.3 Convergence Detection

Combines two signals:
- **Semantic convergence**: Same bottleneck type with no applicable fixes
- **Diminishing returns**: Significant reduction in CodeGen duration

### 3.4 Tool Interceptor Architecture

Blocks unavailable tools (e.g., `run_ncu` when NCU is restricted) with <1ms fast-fail instead of 35s timeout.

## 4. Measurement Methodology

### 4.1 SM Count Detection

Pointer-chasing kernel with concurrent block execution to detect SM count plateau.

### 4.2 DRAM Bandwidth Measurement

Sustained memory copy kernel with volatile qualifiers and `asm volatile` memory barriers to prevent compiler optimization.

### 4.3 Throughput Measurement

Compute-intensive kernels measured via `clock64()` and `cudaEventElapsedTime()` for cross-validation.

### 4.4 Device Attributes

Hardware probing via CUDA runtime API with cross-validation against micro-benchmark results.

***

## 5. Error Handling & Graceful Degradation

When tools are unavailable:
1. **Fast fail**: Return cached error in <1ms
2. **Fallback**: Use alternative measurement methods
3. **Confidence adjustment**: Reduce confidence score for degraded measurements
4. **Version rollback**: Revert to best-quality measurement version

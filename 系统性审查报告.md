## 1. 消息传递链 (Message Passing Chains)

### 1.1 正常路径追踪

**Pipeline -> StageExecutor -> Agent -> SubAgentResult**:

```
Pipeline.run() → PipelineContext → StageExecutor.execute() → CollaborationMessage 
→ BaseSubAgent.execute() → _process() → SubAgentResult → PipelineContext.update()
```

**问题 M1 (严重): AgentLoop 与 Pipeline/StageExecutor 之间的执行模型存在两种互不兼容的路径**

在 [system\_builder.py:L132-157](file:///e:/GPU_Profiling_System/src/application/system_builder.py#L132-L157) 中，`build_pipeline` 方法创建了一个纯 `Pipeline` 对象，其中的 `AgentLoop` 是在 [stage\_executor.py:L302-310](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L302-L310) 中动态创建的。

但在 [agent\_loop.py:L971-1009](file:///e:/GPU_Profiling_System/src/application/agent_loop.py#L971-L1009) 中，`AgentLoop.run_pipeline()` 接受一个 `Pipeline` 对象并在其中调用 `pipeline.run()`。

这意味着**存在两套完全独立的 AgentLoop 实例**：

1. Pipeline 模式下：StageExecutor 为每个阶段创建新的 AgentLoop（每个阶段一个）
2. AgentLoop 模式下：AgentLoop 自己拥有一个独立的 start() 循环

这两个路径的消息传递机制完全不同，上下文不会在两者之间共享。

### 1.2 CollaborationMessage 传递中的信息丢失

**问题 M2 (中等): MetricAnalysis 阶段可能收不到完整的 CodeGen 数据**

在 [stage\_executor.py:L124-148](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L124-L148)，`_build_collaboration_message` 从 `PipelineContext` 中提取 `code_gen_data`，但只在 `step.stage in (METRIC_ANALYSIS, VERIFICATION)` 时包含。

问题在于：

- `ctx.code_gen_data` 在 [pipeline\_context.py:L61-62](file:///e:/GPU_Profiling_System/src/domain/pipeline_context.py#L61-L62) 中只有 `CODE_GEN SUCCESS` 时才被设置
- 如果 CodeGen 部分成功（`_can_continue_with_partial` 返回 True），`ctx.update()` 中的 `code_gen_data` 不会被设置（因为 `result.is_success()` 为 False）
- 这导致 MetricAnalysis 可能拿到空的 CodeGen 数据

### 1.3 Planner -> CodeGen 的 task 信息传递

**问题 M3 (中等): Planner 的 task 分解结果在传递到 CodeGen 时存在路径断裂**

追踪数据流：

1. Planner 产生 `SubAgentResult(data={"tasks": [...]})` — [planner.py:L112-134](file:///e:/GPU_Profiling_System/src/application/subagents/planner.py#L112-L134)
2. Pipeline 通过 `ctx.update(PLAN, result)` 存储 — [pipeline.py:L180](file:///e:/GPU_Profiling_System/src/domain/pipeline.py#L180)
3. 进入 CODE\_GEN 阶段时，`ctx.prev_result` 是 PLAN 的结果
4. StageExecutor.\_build\_collaboration\_message() 提取 `prev_result.data` — [stage\_executor.py:L127-128](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L127-L128)
5. 在 `_run_with_agent_loop` 中：

```python

    tasks_list = prev_result.get("data", {}).get("tasks", [])
    if tasks_list:
        task = {"tasks": tasks_list}
```

这里 `task` 变量是从 `message.payload` 中提取的。但 `message.payload` 在 `_build_collaboration_message` 中构造时：

```python

    payload["prev_result"] = ctx.prev_result.to_dict()
    payload["prev_fingerprint"] = ctx.prev_result.context_fingerprint
```

注意：这里存的是 `ctx.prev_result.to_dict()`，而 `to_dict()` 在 [subagent.py:L46-55](file:///e:/GPU_Profiling_System/src/domain/subagent.py#L46-L55) 中把 `data` 序列化为字典。所以 `prev_result.get("data", {})` 实际拿到的是 `{"data": {"tasks": [...]}, ...}`。

但等等——在 `_run_with_agent_loop` 中：


如果 `payload` 中**没有** `"task"` key（Planner 阶段不会设置它），那么 `task = {}`，然后 fallback 到 `prev_result.get("data", {}).get("tasks", [])`。

**这条链路是能通的**，但代码可读性很差，且容易在修改时断裂。

## 2. 工具调用链 (Tool Calling Chains)

### 2.1 完整链路追踪

```
LLM Output → CompositeToolCallParser.parse() → ToolCall 
→ AgentLoop._execute_tool_call() → _execute_with_approval() → tool_executor 
→ ToolRunner.execute() → SchemaValidator → ApprovalQueue → handler
```

### 2.2 问题 T1 (严重): StageExecutor 中工具执行器与 ToolRunner 的混合使用

在 [stage\_executor.py:L317-320](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L317-L320)：

```python
if self._sandbox and self._tool_handlers:
    handlers = dict(self._tool_handlers)
    loop.set_tool_executor(lambda tool_name, args: handlers[tool_name](args))
```

这里绕过了 `ToolRunner` 的完整验证管线（schema validation, approval check, output validation），直接调用 handler。这意味着：

- **输入 schema 验证被跳过**
- **审批检查被跳过**（除非在 `HIGH_AUTONOMY` 模式下全部自动批准）
- **输出 schema 验证被跳过**

而在 `AgentLoop` 自己的 `_execute_tool_call` 中，它只做了 permission 检查，没有做 schema 验证。

**影响**：在 Pipeline 模式下，LLM 发送的畸形参数会直接进入 handler，没有被 `SchemaValidator` 拦截。

### 2.3 问题 T2 (中等): compile\_cuda\_handler 与 CodeGenAgent.\_compile() 使用不同的路径

`compile_cuda_handler` ([compile\_cuda.py:L102-230](file:///e:/GPU_Profiling_System/src/infrastructure/tools/compile_cuda.py#L102-L230)) 在 Pipeline 模式下通过 tool executor 调用。

但 `CodeGenAgent._compile()` ([codegen.py:L267-323](file:///e:/GPU_Profiling_System/src/application/subagents/codegen.py#L267-L323)) **不使用** `compile_cuda_handler`，而是直接在 agent 内部调用 `self._sandbox.run()`。

这意味着：

- **两套完全不同的编译逻辑**
- Pipeline 模式的 CodeGen stage 用 handler（有 arch auto-detect, flag sanitization）
- 如果 CodeGenAgent 单独运行（非 Pipeline），直接用 `_compile()`（架构检测但无 flag sanitization）
- 两者编译输出路径不同：handler 用 `src/source.cu` + `bin/benchmark`，而 `_compile()` 用 `src/source.cu` + `bin/benchmark_{target}`

**这导致 binary\_path 不一致**——handler 返回的 binary\_path 是 `bin/benchmark`，而 CodeGenAgent 内部的 `_execute` 找的是 `bin/benchmark_{target}`。

### 2.4 问题 T3 (严重): ToolCallParser 的 BraceScanToolCallParser 在 JSON 中包含字符串内的花括号时会失败

[tool\_call\_parser.py:L64-87](file:///e:/GPU_Profiling_System/src/application/tool_call_parser.py#L64-L87)：

`BraceScanToolCallParser` 简单地追踪 `{` 和 `}` 的深度。但 CUDA 代码中包含大量花括号（函数体、控制流）。如果 LLM 把完整的 CUDA 源代码直接放在文本中（而不是 JSON 中），BraceScan 会尝试解析代码块中的第一个 `{` 到匹配的 `}` 之间的内容作为 JSON，这必然失败。

然而这不是致命问题，因为 JsonToolCallParser 优先于 BraceScan，而 LLM 通常输出正确的 JSON。但如果 LLM 输出格式不规范，BraceScan 可能在找到真正的 JSON 之前就返回了一个错误匹配。

### 2.5 问题 T4 (中等): FuzzyToolCallParser 对 `compile_cuda` 的正则表达式可能错误匹配

[tool\_call\_parser.py:L111-126](file:///e:/GPU_Profiling_System/src/application/tool_call_parser.py#L111-L126)：

```python
rf'\b{re.escape(tool_name)}\s*\(([^)]*)\)',
```

对于 `compile_cuda(source="...", flags=["-O3"])`，由于 `source` 参数中包含 `"..."` 可能很长且包含嵌套括号（如 `#include <cuda_runtime.h>`），`[^)]*` 会在第一个 `)` 处停止。这意味着如果 LLM 的输出包含 `compile_cuda(...)` 且参数很长，fuzzy parser 只能截断地提取参数。

## 3. 错误恢复链 (Error Recovery Chains)

### 3.1 编译错误恢复

**路径**：编译失败 → tool result status=error → AgentLoop 添加 guidance → 下一 turn LLM 修复 → 重新编译

**问题 E1 (严重): CodeGen 阶段的编译错误恢复存在双轨制**

有**两层**编译错误恢复机制：

1. **AgentLoop 级别的恢复**（[agent\_loop.py:L267-281, L410-428](file:///e:/GPU_Profiling_System/src/application/agent_loop.py#L267-L281)）：
   - 检测空 source code → 添加 guidance → failure tracking → 终止条件
   - 编译错误 → `_build_compile_error_guidance()` → system guidance
2. **CodeGenAgent 内部的恢复**（[codegen.py:L105-131](file:///e:/GPU_Profiling_System/src/application/subagents/codegen.py#L105-L131)）：
   - `max_compile_retries = 3` 的 while 循环
   - 编译失败 → 添加错误信息到 context → LLM 重新生成

**问题**：在 Pipeline 模式下，这两个机制同时运行！StageExecutor 创建 AgentLoop，AgentLoop 有自己的错误处理。同时 CodeGenAgent 的 `_process()` 方法内部有独立的编译重试。但实际执行时：

- Pipeline 模式：StageExecutor 创建的 AgentLoop 使用 `agent.context_manager`，LLM 在这个 context manager 上调用。CodeGenAgent 的 `_process` 被调用的路径是通过 AgentLoop 的工具调用，**而不是**直接调用 `_process`。
- 所以 CodeGenAgent 内部的编译重试循环**只在非 Pipeline 模式下生效**（当 CodeGenAgent 被直接 `.execute()` 调用时）。

在 Pipeline 模式下，编译错误恢复**只依赖 AgentLoop 的 guidance 机制**——它告诉 LLM 修复代码并重试。这个机制是有效的，但依赖 LLM 自行理解错误并修复。

### 3.2 空编译/空执行的 anti-loop 保护

**问题 E2 (中等): Anti-loop 保护的计数可能不准确**

[agent\_loop.py:L271-279](file:///e:/GPU_Profiling_System/src/application/agent_loop.py#L271-L279)：

```python
empty_compile_pattern = "compile_cuda_empty_source"
self._failure_tracker.record_failure(empty_compile_pattern)
if self._failure_tracker.should_terminate(empty_compile_pattern):
```

`InvariantTracker.should_terminate()` 的阈值是硬编码的。如果阈值设置过低（比如 2），LLM 可能只犯了两次同样的错误就被终止。如果过高（比如 5），会浪费大量 token 和 API 调用。

### 3.3 Verification 拒绝后的重试

**问题 E3 (中等): 重试时 CodeGen 的 context 被重置，但 conversation\_history 可能包含冲突信息**

在 [pipeline.py:L163-168](file:///e:/GPU_Profiling_System/src/domain/pipeline.py#L163-L168)：

```python
ctx.prev_result = None
ctx.prev_stage = None
continue
```

当 Verification 拒绝后，Pipeline 跳回 CodeGen。`ctx.prev_result` 和 `ctx.prev_stage` 被重置。但：

1. StageExecutor 创建**新的** AgentLoop，使用**新的** `agent.context_manager`
2. `_build_retry_message` 从 `ctx.get_feedback_for_codegen()` 提取拒绝信息
3. 但 CodeGenAgent 的 context\_manager 是**全新的**——它不包含之前尝试的任何测量数据

在 [stage\_executor.py:L186-267](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L186-L267) 中，`_build_retry_message` 包含了详细的拒绝反馈、MetricAnalysis 反馈、以及 zero measurement 修复指导。**这部分设计是好的**——所有必要的上下文都在 retry message 的 payload 中传递。

但是 `_save_conversation_history` ([stage\_executor.py:L345-390](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L345-L390)) 保存的对话历史被附加到 `ctx.conversation_history`，而重试时会通过 `_format_conversation_history` 注入到 prompt 中。**问题是**：对话历史中包含旧的（被拒绝的）测量数据，可能与新的测量冲突。

## 4. 上下文管理链 (Context Management Chains)

### 4.1 问题 C1 (严重): ContextManager.compress() 可能丢失关键测量数据

[context.py:L302-376](file:///e:/GPU_Profiling_System/src/application/context.py#L302-L376)

压缩策略：

1. Phase 1: 移除 DISPOSABLE
2. Phase 2: 摘要化 LOW
3. Phase 3: 摘要化 MEDIUM
4. Phase 4: 移除 MEDIUM

问题在于 priority 分类。在 [\_classify\_priority](file:///e:/GPU_Profiling_System/src/application/context.py#L43-L102) 中：

- 成功的 tool output → `Priority.HIGH`
- 错误消息 → `Priority.MEDIUM`
- Control Plane → `Priority.LOW`
- 短的自然语言回复 → `Priority.DISPOSABLE`

**关键问题**：如果 LLM 在一个 turn 中输出自然语言 + tool call，自然语言部分被添加为 ASSISTANT 角色。如果这段自然语言很短（< 100 字符），它被分类为 DISPOSABLE，在压缩时首先被删除。

但如果这段短文本包含关键信息（比如 "I'll measure dram\_latency using pointer-chasing"），它被删除后，LLM 在下一次迭代中可能丢失之前的意图。

### 4.2 问题 C2 (严重): token\_count 估算方式不准确

[context.py:L39-40](file:///e:/GPU_Profiling_System/src/application/context.py#L39-L40)：

```python
if self.token_count <= 0:
    self.token_count = max(1, len(self.content) // 3 + 1)
```

这里用的是 `len(content) // 3`——即 3 字符/token。但：

- 在 [context.py:L105-130](file:///e:/GPU_Profiling_System/src/application/context.py#L105-L130) 的 `_estimate_tokens` 中，估算逻辑是：
  - 特殊字符比率 > 0.15 → 2.5 字符/token
  - 特殊字符比率 < 0.05 → 4.0 字符/token
  - 中间值 → 3.0 字符/token

**不一致**：`ContextEntry.__post_init__` 使用固定的 `// 3`，而 `_estimate_tokens` 使用动态比率。但 `add_entry` 调用的是 `_estimate_tokens`，而 `update_system_entry` 直接使用传入的 `token_count` 参数（通常很小，如 50）。

**实际影响**：`update_system_entry` 中 Control Plane 的 token\_count 被设为 50，但实际渲染后的文本可能有 200-500 字符。这意味着 `is_over_budget` 检查严重低估了 Control Plane 的 token 消耗。

### 4.3 问题 C3 (中等): `to_messages()` 不处理 `content` 为 None 的情况

[context.py:L382-386](file:///e:/GPU_Profiling_System/src/application/context.py#L382-L386)：

```python

    return [
        {"role": e.role.value, "content": e.content if e.content is not None else ""}
```

这一行处理了 `None`。但 model\_caller 中也有类似的检查。不过如果 `e.content` 是一个非常长的字符串（比如 10000+ 字符的 CUDA 代码），`to_messages` 不做任何截断——完整的超长内容会被发给 LLM API。

## 5. 流水线交接链 (Pipeline Handoff Chains)

### 5.1 问题 H1 (严重): Pipeline 阶段之间的数据格式契约不一致

**PLAN -> CODE\_GEN 交接**：

- PLAN 输出：`SubAgentResult(data={"tasks": [...], "targets": [...], "plan": [...]})`
- CODE\_GEN 期望：从 `prev_result.data` 中提取 `tasks`，然后匹配当前 target

在 [stage\_executor.py:L282-285](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L282-L285)：

```python
if not task and stage_name == PipelineStage.CODE_GEN.value:
    tasks_list = prev_result.get("data", {}).get("tasks", [])
    if tasks_list:
        task = {"tasks": tasks_list}
```

这里 `prev_result` 是 `payload.get("prev_result", {})`，是 `ctx.prev_result.to_dict()` 的副本。`to_dict()` 序列化 `data` 为字典。所以 `prev_result.get("data", {})` 返回的是 Planner 的 `data` 字典，其中包含 `tasks` 列表。

**这条链是通的**。

**CODE\_GEN -> METRIC\_ANALYSIS 交接**：

- CODE\_GEN 输出：`SubAgentResult(data={"measurements": {...}, "tool_results": [...], "final_output": "...", ...})`
- METRIC\_ANALYSIS 期望：从 `prev_result.data` 或 `ctx.code_gen_data` 获取测量值

在 [stage\_executor.py:L134-137](file:///e:/GPU_Profiling_System/src/domain/stage_executor.py#L134-L137)：

```python
if ctx.code_gen_data and step.stage in (
    PipelineStage.METRIC_ANALYSIS, PipelineStage.VERIFICATION
):
    payload["codegen_data"] = ctx.code_gen_data
```

**但注意**：`ctx.code_gen_data` 只有在 `result.is_success()` 时才被设置（[pipeline\_context.py:L61](file:///e:/GPU_Profiling_System/src/domain/pipeline_context.py#L61)）。如果 CodeGen 返回 FAILED 但 `_can_continue_with_partial` 为 True，Pipeline 继续但不设置 `code_gen_data`。MetricAnalysis 阶段拿不到任何 CodeGen 数据。

### 5.2 问题 H2 (严重): Pipeline.retry 时阶段索引重置存在逻辑错误

[pipeline.py:L126-178](file:///e:/GPU_Profiling_System/src/domain/pipeline.py#L126-L178)：

```python
if result.status == SubAgentStatus.REJECTED and step.stage == PipelineStage.VERIFICATION:
    ...
    if ctx.can_retry():
        ctx.increment_iteration()
        code_gen_idx = self._find_stage_index(PipelineStage.CODE_GEN)
        if code_gen_idx is not None:
            stage_idx = code_gen_idx
            ctx.prev_result = None
            ctx.prev_stage = None
            continue
```

**问题**：当 retry 跳回 CODE\_GEN 阶段后：

1. `stage_idx = code_gen_idx`（通常是 1）
2. `ctx.prev_result = None` — 但 CODE\_GEN 阶段需要知道 PLAN 的结果（包含 tasks）
3. 下一次循环进入 CODE\_GEN 时，`ctx.prev_result` 为 None
4. `_build_collaboration_message` 检查 `ctx.prev_result`——它是 None
5. 因此 `payload["prev_result"]` 不会被设置
6. CODE\_GEN 阶段**丢失了 Planner 的 task 分解结果**

**这是一个数据丢失的 bug**。虽然 `_build_retry_message` 中的 feedback payload 包含了 `prev_result` 的替代数据，但它只在 `feedback` 存在且 `step.stage == CODE_GEN` 时触发：

```python
# stage_executor.py:L86-90
feedback = ctx.get_feedback_for_codegen()
if feedback and step.stage == PipelineStage.CODE_GEN:
    message = self._build_retry_message(step, ctx, feedback)
else:
    message = self._build_collaboration_message(step, ctx)
```

当 retry 时，`ctx.rejection_history` 有内容（因为 `add_rejection` 被调用了），所以 `get_feedback_for_codegen()` 返回非 None。因此走 `_build_retry_message` 路径。

在 `_build_retry_message` 中：

```python
if ctx.code_gen_data:
    payload["prev_result"] = {
        "agent_role": AgentRole.CODE_GEN.value,
        "status": "rejected_retry",
        "data": ctx.code_gen_data,
    }
```

所以 retry 时使用的是 `ctx.code_gen_data` 而不是 `ctx.prev_result`。但 `ctx.code_gen_data` 是上一次 CodeGen 的结果（被拒绝的那个）。它包含 `measurements`、`tool_results` 等。

**关键缺失**：`_build_retry_message` 中的 payload **没有包含原始的 Planner tasks**。`_build_retry_message` 设置了 `payload["target_spec"]`（如果 `ctx.target_spec` 存在），但没有设置 tasks。

而在 `_run_with_agent_loop` 中：

```python
if not task and stage_name == PipelineStage.CODE_GEN.value:
    tasks_list = prev_result.get("data", {}).get("tasks", [])
```

这里 `prev_result` 是 `payload.get("prev_result", {})`，在 retry 中是 `{"agent_role": "code_gen", "status": "rejected_retry", "data": ctx.code_gen_data}`。所以 `prev_result.get("data", {}).get("tasks", [])` 会查看 `ctx.code_gen_data.get("tasks", [])`——但 CodeGen 的 data 中**没有** `tasks` key！

**结论**：retry 时 CODE\_GEN 阶段**拿不到 Planner 的 tasks**。它只能拿到上一次 CodeGen 的测量数据和拒绝反馈。这意味着 retry 时 LLM 不知道每个 target 对应什么 measurement method。

```python
def _already_executed_binary(self, binary_path: str) -> bool:
    entries = self.context_manager.get_entries()
    last_compile_idx = -1
    
    # 找到最后一次编译的位置
    for i, entry in enumerate(entries):
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if data.get("tool") == "compile_cuda" and data.get("success") is True:
                    last_compile_idx = i
            except: pass
    
    if last_compile_idx == -1:
        return False  # 没有编译过
    
    # 检查最后一次编译后是否有执行
    for entry in entries[last_compile_idx + 1:]:
        if entry.role.value == "assistant":
            try:
                data = json.loads(entry.content)
                if data.get("tool") == "execute_binary" and data.get("binary_path") == binary_path:
                    return True
            except: pass
    
    return False
```

**问题**: DISPOSABLE 级别（短自然语言回复 < 100 字符）在压缩时首先被删除。但如果这些短回复包含关键意图信息（如"我将使用 pointer-chasing 测量 DRAM 延迟"），删除后 LLM 可能丢失上下文。

***

### 🟡 P2-2: token\_count 估算不一致

**文件**: `context.py:L39-40` vs `context.py:L105-130`

**问题**:

- `ContextEntry.__post_init__`: 固定 `len(content) // 3`
- `_estimate_tokens`: 动态比率（2.5\~4.0 字符/token）
- `update_system_entry`: 直接使用传入的 `token_count`（通常很小，如 50）

**影响**: `is_over_budget` 检查不准确，可能导致上下文超出 LLM 的 token 限制。

***

### 🟡 P2-3: 状态持久化不完整

**文件**: `agent_loop.py:L913-922`

**问题**: 只保存 `LoopState` 和 `SessionState`，**不保存**:

- `context_manager._entries`（对话上下文）
- `control_plane._progress`
- `failure_tracker` 的状态

**影响**: 崩溃后无法完全恢复，LLM 需要从头开始。

***

## 关键链路逐一审查

### 链路 1: Planner → CodeGen 数据传递

```
Planner 输出 JSON tasks
  → SubAgentResult(data={"tasks": [...]})
  → ctx.update(PLAN, result)
  → ctx.prev_result = result
  → CODE_GEN 阶段
  → _build_collaboration_message
  → payload["prev_result"] = ctx.prev_result.to_dict()
  → _run_with_agent_loop
  → task = prev_result.get("data", {}).get("tasks", [])
```

**审查结果**: ✅ **正常**（首次运行）
**审查结果**: ❌ **异常**（retry 时，见 P0-2）

***

### 链路 2: CodeGen → MetricAnalysis 数据传递

```
CodeGen 输出 measurements + tool_results
  → SubAgentResult(data={"measurements": {...}, "tool_results": [...]})
  → ctx.update(CODE_GEN, result)
  → ctx.code_gen_data = result.data (仅当 is_success())
  → METRIC_ANALYSIS 阶段
  → payload["codegen_data"] = ctx.code_gen_data
```

**审查结果**: ⚠️ **部分有效**

- 如果 CodeGen `is_success()` → 正常
- 如果 CodeGen FAILED 但 `_can_continue_with_partial` 为 True → `code_gen_data` 不被设置 → MetricAnalysis 没有数据

***

### 链路 3: compile\_cuda → execute\_binary 工具调用链

```
LLM 输出 compile_cuda JSON
  → JsonToolCallParser 解析
  → ToolCall(name="compile_cuda", arguments={...})
  → AgentLoop._execute_tool_call()
  → _execute_with_approval()
  → tool_executor (handlers[tool_name])
  → compile_cuda_handler(source, flags)
  → nvcc 编译
  → 返回 {binary_path: "...", success: true}
  → context_manager.add_entry(ASSISTANT, result_str)
  → 添加编译成功引导
  → 下一轮 LLM 看到引导 → execute_binary
```

**审查结果**: ⚠️ **部分断裂**

- 编译成功引导有效（Turn 2 成功）
- 但 Turn 4 断裂（见 P1-2, P1-3）

***

### 链路 4: LLM 编译错误 → 自我修复链

```
LLM 编译失败
  → tool result status="error"
  → AgentLoop 添加 guidance (_build_compile_error_guidance)
  → 下一轮 LLM 看到 guidance
  → LLM 修复代码
  → 重新 compile_cuda
```

**审查结果**: ⚠️ **依赖 LLM 能力**

- 引导消息提供精确修复指导（✅）
- 但 LLM 可能仍然无法修复（LLM 能力问题）
- 没有 fallback 路径（P2 Harness Engineering 设计）

***

### 链路 5: Verification REJECT → CodeGen Retry 链

```
Verification REJECT
  → ctx.add_rejection(...)
  → ctx.can_retry() → True
  → ctx.increment_iteration()
  → stage_idx = code_gen_idx
  → ctx.prev_result = None  # ← 数据丢失！
  → ctx.prev_stage = None
  → continue (跳回 CODE_GEN)
  → _build_retry_message (因为有 rejection feedback)
  → payload["prev_result"] = ctx.code_gen_data  # 只有上一次 CodeGen 数据
  → CodeGen 收到 retry message
```

**审查结果**: ❌ **数据丢失**（见 P0-2）

***
